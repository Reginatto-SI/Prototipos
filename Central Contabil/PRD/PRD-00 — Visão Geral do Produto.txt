# PRD-00 — Visão Geral do Produto
# Central Contábil

---

## 1. Objetivo do Sistema

O Central Contábil é uma plataforma SaaS multi-tenant voltada para escritórios de contabilidade, com o objetivo de centralizar todas as informações operacionais dos clientes em um único sistema simples, rápido e confiável.

O sistema deve substituir:
- planilhas
- blocos de notas
- pastas de rede (arquivos compartilhados)

A proposta é funcionar como uma “plataforma operacional contábil”, sendo o principal ponto de consulta e execução da equipe no dia a dia.

---

## 2. Princípios do Produto

### 2.1 Simplicidade operacional
- Interface direta e objetiva
- Evitar fluxos complexos
- Priorizar ações rápidas

---

### 2.2 Centralização total
- O usuário não deve precisar sair do sistema para buscar informações
- Todas as informações relevantes do cliente devem estar disponíveis no sistema

---

### 2.3 Sistema ativo (não passivo)
- O sistema deve avisar, sugerir e organizar o trabalho
- Não depender apenas da ação manual do usuário

---

### 2.4 Performance
- Respostas rápidas (<200ms sempre que possível)
- Carregamento assíncrono
- Evitar telas pesadas

---

### 2.5 Reutilização de padrões
- Não criar novos padrões se já existir um equivalente
- Manter consistência visual e funcional

---

## 3. Arquitetura Conceitual

### 3.1 Multi-tenant

O sistema é multi-tenant por escritório contábil.

- Isolamento obrigatório por `office_id`
- Nenhum dado pode ser acessado entre escritórios diferentes

---

### 3.2 Cliente como entidade central (HUB)

O cliente é a entidade principal do sistema.

Todas as informações operacionais devem estar organizadas dentro do cliente.

Não deve existir informação operacional relevante fora do contexto de um cliente.

---

### 3.3 Tipos de Cliente

O sistema deve suportar:

- Pessoa Física (PF)
- Pessoa Jurídica (PJ)

#### Regras:
- O tipo define comportamento da interface
- Documento deve suportar CPF e CNPJ
- Busca deve funcionar com ou sem formatação

---

### 3.4 Perfil Operacional/Fiscal do Cliente

Além do tipo de cliente (PF/PJ), o sistema deve suportar um perfil operacional/fiscal para permitir regras de automação, geração de obrigações e filtros de gestão.

Perfis iniciais suportados:

- Lucro Real
- Lucro Presumido
- Simples Nacional
- MEI
- Produtor Rural

#### Regras:
- o perfil operacional/fiscal não substitui PF/PJ
- PF/PJ define a natureza do cliente
- o perfil operacional/fiscal define comportamento das obrigações e automações
- esse campo deve ser usado como base para:
  - geração automática de obrigações
  - aplicabilidade de regras
  - filtros operacionais
  - futuras automações do sistema

#### Observação:
- a lista de perfis poderá ser expandida no futuro, mas esses são os perfis oficiais iniciais do sistema

## 4. Estrutura Geral do Sistema

---

### 4.1 Clientes (Módulo Principal)

O cliente funciona como HUB centralizador.

Abas internas:
- Dados gerais
- Acessos
- Certificados digitais
- Anotações
- Documentos
- Extratos bancários

---

### 4.2 Obrigações (Módulo Operacional)

Responsável por controlar tarefas contábeis.

Características:
- geração por cliente + competência
- controle por status
- prazos internos e fiscais
- responsável por execução

---

### 4.3 Automação Operacional

Responsável por:

- gerar obrigações automaticamente
- definir responsáveis e prazos
- atualizar indicadores de atraso

---

### 4.4 Notificações e Alertas

Responsável por:

- avisar sobre prazos próximos
- alertar atrasos
- informar mudanças relevantes

---

### 4.5 Dashboard Operacional

Responsável por:

- mostrar o que precisa ser feito
- destacar prioridades
- permitir ação rápida sobre obrigações

---

### 4.6 Usuários e Permissões

Controle de acesso baseado em:

- usuário
- carteira de clientes

Tipos:
- gerente (acesso total)
- usuário (acesso restrito)

---

### 4.7 Busca Global

Permite localizar rapidamente:

- clientes
- obrigações
- documentos
- acessos

A busca deve considerar:
- nome
- documento
- descrição de documentos
- palavras-chave

---

### 4.8 Auditoria e Histórico

Responsável por:

- registrar alterações
- rastrear ações
- garantir transparência

---

## 5. Regras de UX

### 5.1 Interface
- simples e direta
- sem excesso de informação

---

### 5.2 Interações
- ações principais visíveis
- ações secundárias via menu “...”

---

### 5.3 Produtividade
- copiar dados com 1 clique
- acesso rápido a documentos
- minimizar cliques

---

## 6. Diretrizes Técnicas (Alto Nível)

### 6.1 Estrutura de dados
- todas as entidades devem conter `office_id`
- isolamento obrigatório

---

### 6.2 Padrões
- snake_case
- CRUD completo
- paginação obrigatória
- filtros reutilizáveis

---

### 6.3 Segurança
- controle por usuário
- controle por carteira
- proteção de dados sensíveis

---

## 7. Fora de Escopo

O sistema NÃO deve:
- se tornar um ERP
- realizar apuração fiscal
- substituir sistemas contábeis

---

## 8. Regra de Ouro

Sempre que houver dúvida:

→ escolher a solução mais simples, clara e rápida para o usuário.
